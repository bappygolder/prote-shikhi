- ⚠️ **Non-blocking deviation — freshness term**. Spec §8 specifies `min(card.cardsAgoSeen, 5) * W_FRESHNESS`, where `cardsAgoSeen` is a per-card session counter bumped on every other card's grade. The implementation uses `(5 - min(recent.length, 5)) * W_FRESHNESS` as a stand-in ([lib/learning.ts:459-466](../../lib/learning.ts#L459-L466) — the comment block names this explicitly). Effect is opposite-direction from spec: a card with no recent results gets a high freshness term (5 × 0.5 = 2.5), a card with 5+ recent results gets 0. Spec wants "haven't seen this card lately → bump"; the proxy gives "this card has fewer recent grades → bump." For new entrants both readings agree (both push toward "show me"). For mid-rotation cards the readings can disagree, but the magnitude is small (cap is 2.5 weight units against W_RECENT_MISS=4 + boost up to 8 + penalty term). **Not a FAIL** because (a) the CTX-06 prompt does not require `cardsAgoSeen` plumbing and the spec puts that bookkeeping in CTX-07 / sprinkle territory ("freshness — nudges variety; falls off after a few cards"), (b) the comment in code transparently names it as a CTX-06 approximation, and (c) it does not alter any of the hard guarantees the manual-scenario checklist tests for. Worth tightening in CTX-07 when full session-distance bookkeeping lands. Logged as observation #1 below.

## Findings (non-blocking — none are FAIL)

1. **Freshness term is a stand-in, not the spec's `cardsAgoSeen`.** Code uses `(5 - min(recent.length, 5)) * W_FRESHNESS`, comment names it explicitly as a CTX-06 approximation. The proxy is bounded the same way (≤ 2.5 weight units) but the semantic flips for mid-rotation cards. CTX-07 should plumb a real per-card `cardsAgoSeen` counter (or fold it into `SessionState`) so this term matches the spec, especially since the sprinkle eligibility logic in CTX-07 is going to need real session-distance bookkeeping anyway.

2. **`applyActiveSetOnCorrect` does not validate the `cardId`.** The function ignores its `_cardId` and `_cardProgress` arguments and trusts the caller to invoke it only on a first counted-correct of an active card. App.tsx's `wasFirstCountedCorrect` gate ([App.tsx:750-753](../../App.tsx#L750-L753)) provides that check, so the contract holds — but the asymmetry with `applyActiveSetOnMastery` (which DOES use `masteredCardId`) is a small papercut. Fine as-is; worth a one-line doc comment in CTX-07.

3. **Newcomer entrant gets `attemptsSinceEnteringActive=0` *and* gets bumped to 1 the same tick if it's the chosen card.** This is the spec-correct outcome — the bump in [App.tsx:838-847](../../App.tsx#L838-L847) corresponds to spec §10's "After picking … chosen.attemptsSinceEnteringActive += 1." Just calling it out so a future reader doesn't suspect a double-count. Net: entrant carries the boost for its first 8 *displays*, not its first 9.

4. **Other practice lists (`needsWork`, `mastered`, `সব`) bypass `SessionState`.** The implementation passes them through `chooseNextCard` without a `session`, so they hit `transientSession`, which builds a one-shot active set and runs the same scoring. This means anti-immediate-repeat still applies on those lists too (good), but struggle mode and active-set growth do not (intentional — those lists aren't path-driven). Spec §3 / §9 don't define behavior for those lists, so this is consistent. Document in CTX-07 if the spec ever extends to teacher-overrides.

5. **Manual browser scenario was not interactively performed** — port 8081 was busy on this machine. Mitigated by the unit-test coverage matrix above. Bappy may want to verify in a fresh browser tab when convenient: reset progress → consonants preset → grade ✓ ✗ ✓ ✓ ✗ ✗ ✓ ✓ ✓ ✓ ✓ ✓ → confirm the behavioral patterns in the prompt's manual-scenario list.
